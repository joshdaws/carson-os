# `_disagreements.md` — hand-mocked example

SPIKE deliverable #4 (v5 design doc, "The Assignment" §4). Hand-mocked for the
Daws household to test whether the contradiction-surfacing pattern actually
feels useful at family scale, not just trivially "Becca said yes, Josh said
no."

In production this file lives at `~/.carsonos/memory/household/_disagreements.md`
and is regenerated nightly by the v5.1 compilation agent from atoms across all
collections. **It is a compiled view, not a memory.** If something here is wrong,
the fix is to append a corrective atom via `correct_memory` — the next
compilation pass will clear or update the entry.

The example below shows seven entries that span the realistic shapes of
family contradiction: parent-to-parent coordination drift, household state
that decays without reconciliation, calendar conflicts, preference evolution,
authority disagreement between trusted sources, soft contradictions worth
auditing, and likely-resolved entries kept for one cycle as a verification
prompt.

## What this hand-mock is testing

The SPIKE deliverable asks whether the contradiction-surfacing pattern feels
useful at family scale. Reading the seven entries above and noting what's
genuine vs. contrived:

- **Genuinely useful.** Bedtime, spare key, vacation week, reading
  assessment, and medication list all describe real household-coordination
  problems where surfacing the conflict beats smoothing it. The audit
  prompt for the spare key especially — a family without this view would
  only notice the contradiction during an actual lockout.
- **Worth surfacing as soft conflict.** The Hudson coffee-shop preference
  shift and the Wonderland snacks resolution are not strict contradictions
  but are useful to keep visible for one cycle as the agent learns the
  family's pace of preference change.
- **Reveals a schema gap.** The reading-level entry surfaced a need for a
  `concept` memory (methodology) — the contradiction is a SYMPTOM of the
  family treating two different assessments as one quantity. The
  disagreement view did real work by exposing that.
- **Format gap.** I needed `importance` to weight the spare-key entry
  (Becca's atom is higher importance because she's more recent + cited a
  specific landmark). Importance is in atom frontmatter per CEO cherry-pick
  #1. Compilation agent should reference it when ordering disagreements.
- **Status taxonomy.** "Unresolved (N days)", "soft conflict", "likely
  resolved" are the three I reached for naturally. Worth formalizing as an
  enum the compilation agent picks from. **Add to design doc Phase 2 spec.**

The pattern feels useful and surfaces non-obvious work. Ship as designed.
